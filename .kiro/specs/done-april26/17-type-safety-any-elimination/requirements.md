# Requirements Document

## Introduction

Replace `any` type annotations in route handler files with proper Prisma-generated types and explicit interfaces. The backend route files `robots.ts`, `admin.ts`, and `user.ts` contain numerous `any` casts, many suppressed with `eslint-disable-next-line @typescript-eslint/no-explicit-any`. This undermines TypeScript's strict mode and hides potential runtime errors.

## Glossary

- **Prisma Generated Types**: TypeScript types auto-generated by Prisma from the database schema, located in `generated/prisma/`. These include model types (e.g., `Robot`, `User`, `Battle`) and input/output types for queries.
- **eslint-disable**: An inline comment that suppresses a specific ESLint rule. Each instance of `@typescript-eslint/no-explicit-any` represents a known type safety gap.

## Expected Contribution

This spec targets the "type safety erosion" debt where `any` types bypass TypeScript's type checking in critical route handlers.

1. **robots.ts any reduction**: From 15+ `eslint-disable @typescript-eslint/no-explicit-any` comments to 0. Before: functions like `sanitizeRobotForPublic(robot: any): any` and battle stat helpers use `any` throughout. After: all typed with Prisma models.
2. **admin.ts any reduction**: From 5+ `eslint-disable` comments to 0. Before: `mapBattleRecord`, `mapTagTeamRecord`, and participant finders use `any`. After: typed with Prisma models and explicit interfaces.
3. **user.ts any reduction**: From 10+ `eslint-disable` comments to 0. Before: cycle snapshot metrics cast to `any[]` with `any` callbacks. After: typed with snapshot interfaces.
4. **Type error prevention**: Before: type mismatches in these files are invisible at compile time. After: TypeScript catches field name typos, missing properties, and incorrect types.

### Verification Criteria

1. `grep -c "eslint-disable.*no-explicit-any" app/backend/src/routes/robots.ts` returns 0
2. `grep -c "eslint-disable.*no-explicit-any" app/backend/src/routes/admin.ts` returns 0
3. `grep -c "eslint-disable.*no-explicit-any" app/backend/src/routes/user.ts` returns 0
4. `grep -c ": any" app/backend/src/routes/robots.ts` returns 0
5. `grep -c ": any" app/backend/src/routes/admin.ts` returns 0
6. `grep -c ": any" app/backend/src/routes/user.ts` returns 0
7. All backend tests pass: `cd app/backend && npm test`
8. TypeScript compilation succeeds with no errors: `cd app/backend && npx tsc --noEmit`

## Requirements

### Requirement 1: Robot Route Type Safety

**User Story:** As a backend developer, I want all functions in the robot route file to use proper types instead of `any`, so that TypeScript catches type errors at compile time.

#### Acceptance Criteria

1. WHEN `sanitizeRobotForPublic` is typed, IT SHALL accept a Prisma `Robot` type (or a specific include type) and return a typed object with sensitive fields omitted.
2. WHEN battle stat helpers (`getBattleResult`, `getBattleStats`) are typed, THEY SHALL accept typed battle objects with participant includes instead of `any`.
3. WHEN ranking helpers (`calculateCategorySum`, `calculateRanking`) are typed, THEY SHALL accept typed parameters instead of `any`.
4. WHEN the `.toNumber()` calls on Decimal fields are typed, THEY SHALL use Prisma's `Decimal` type instead of casting to `any`.
5. WHEN all `any` types are removed, THERE SHALL be zero `eslint-disable @typescript-eslint/no-explicit-any` comments in `robots.ts`.

### Requirement 2: Admin Route Type Safety

**User Story:** As a backend developer, I want all functions in the admin route file to use proper types instead of `any`.

#### Acceptance Criteria

1. WHEN `mapBattleRecord` and `mapTagTeamRecord` are typed, THEY SHALL accept Prisma query result types with explicit includes instead of `any`.
2. WHEN participant finder lambdas are typed, THEY SHALL use the Prisma `BattleParticipant` type instead of `(p: any)`.
3. WHEN all `any` types are removed, THERE SHALL be zero `eslint-disable @typescript-eslint/no-explicit-any` comments in `admin.ts`.

### Requirement 3: User Route Type Safety

**User Story:** As a backend developer, I want all functions in the user route file to use proper types instead of `any`.

#### Acceptance Criteria

1. WHEN cycle snapshot metrics are accessed, THEY SHALL use typed interfaces for `stableMetrics` and `robotMetrics` instead of casting to `any[]`.
2. WHEN reduce/find callbacks operate on metrics, THEY SHALL use the typed metric interface instead of `(m: any)`.
3. WHEN all `any` types are removed, THERE SHALL be zero `eslint-disable @typescript-eslint/no-explicit-any` comments in `user.ts`.

### Requirement 4: No Behavioral Changes

**User Story:** As a user of the API, I want the type safety improvements to produce zero changes to API behavior.

#### Acceptance Criteria

1. WHEN types are added or changed, THE runtime behavior and API responses SHALL remain identical.
2. WHEN all type changes are complete, ALL existing backend tests SHALL pass without modification.
